## Part B: Round 2 추가 이슈

### 이슈 1: [HIGH] AgentDetailView 실시간 갱신 안됨 — 활성 에이전트 상세가 정적 스냅샷

**사용자 시나리오**: 사용자가 활성 에이전트(녹색 점)를 펼치면 Recent Messages, Files, Tools가 표시된다. 하지만 이 데이터는 최초 로드 이후 절대 갱신되지 않는다. 에이전트가 계속 작업 중이어도 "Recent (10)" 메시지 목록은 처음 펼친 순간의 스냅샷에 머문다. 접었다 펴도 캐시된 데이터가 그대로 반환된다.

**위치**: `Sources/ClaudeCodeMonitor/DataLayer/ClaudeDataStore.swift:186`

**근거**:
- `loadAgentDetail()` line 186: `if agentDetailData[key] != nil { return }` — 캐시 히트 시 즉시 반환, 갱신 없음
- `refreshExpandedActiveSessions()` line 85-90: expandedSessionData(에이전트 목록/토큰)만 갱신, agentDetailData는 건드리지 않음
- AgentRow에서 collapse→expand 시 `isExpanded.toggle()` + `loadAgentDetail()` 호출하지만 캐시 히트로 스킵됨

**실제 영향**: 현재 세션의 3개 활성 에이전트(dev, review, qa)가 계속 작업 중. AgentRow의 토큰/도구 카운트는 5초마다 갱신되지만, 펼쳤을 때의 Recent Messages/Files는 최초 로드 시점의 데이터.

**개선 제안**: 활성 에이전트의 agentDetailData를 refreshExpandedActiveSessions()에서 함께 갱신. 또는 `loadAgentDetail`에 `forceRefresh` 파라미터 추가하고, 활성 에이전트 갱신 시 사용.

### 이슈 2: [MEDIUM] 토큰 표시에서 Cache가 In/Out을 압도 — 정보 왜곡

**사용자 시나리오**: 세션 확장 시 토큰 요약이 `[In 2.1M] [Out 549K] [Cache 156M]` 형태로 3개 캡슐이 나란히 표시된다. Cache가 전체의 95%+를 차지하면서 세 캡슐이 동일한 시각적 비중을 가져, 실제 비용에 중요한 In/Out이 시각적으로 동등해 보인다. 또한 cache_read(저렴)와 cache_write(비용 발생)를 합산하여 비용 판단에 도움이 되지 않는다.

**위치**: `Sources/ClaudeCodeMonitor/Views/SessionDetailView.swift:22-28`

**근거**: 실제 에이전트 데이터 — agent-a06835029c34b396e: input=45,618 / output=11,686 / cache_read=1,450,748 / cache_write=1,873,975. Cache 합산(3.3M)이 In+Out(57K)의 58배.

**개선 제안**:
- cache_read와 cache_write를 분리 표시 (비용 차이가 큼)
- 또는 Cache 표시를 아예 제거하고 In/Out만 강조. Cache는 tooltip이나 상세 펼침에서 표시
- 최소한: In/Out을 .caption, Cache를 .caption2로 시각적 위계 구분

### 이슈 3: [MEDIUM] description 없는 에이전트 구별 불가 — 23개 dev가 전부 동일하게 보임

**사용자 시나리오**: 완료 에이전트 목록에서 dev 타입이 23개인데, description이 없어서 모두 `[dev] 3.4M 🔧47` 형태로 표시된다. 토큰 수와 도구 수만으로는 "이 dev가 무엇을 했는지" 알 수 없다. 사용자가 특정 에이전트의 작업 내용을 찾으려면 23개를 하나씩 펼쳐봐야 한다.

**위치**: `Sources/ClaudeCodeMonitor/Views/AgentRow.swift:41-46`

**근거**: 47개 meta.json 중 description이 있는 것은 4개뿐 (Explore 3개, general-purpose 1개). dev(23), qa(10), review(9)는 전부 description 없음.

**개선 제안**: SubagentLoader에서 JSONL의 첫 user 메시지(teammate-message)에서 요약을 추출하여 description이 없을 때 fallback으로 사용. 또는 에이전트가 가장 많이 사용한 도구 상위 3개를 인라인으로 표시 (이미 toolBreakdown이 로드되어 있음).

### 이슈 4: [MEDIUM] scanRecentSessions가 6,000+ JSONL 파일을 매 30초 전수 순회

**사용자 시나리오**: 직접적 UX 영향은 아직 체감되지 않지만 (59ms 소요), `~/.claude/projects/-/` 디렉토리에 6,002개의 JSONL 파일이 존재. `scanRecentSessions()`가 매 30초마다 모든 프로젝트 디렉토리의 모든 JSONL 파일에 `contentsOfDirectory` + `stat()`을 호출. 세션이 늘어나면 선형 증가.

**위치**: `Sources/ClaudeCodeMonitor/DataLayer/ClaudeDataStore.swift:120-154`

**근거**: `~/.claude/projects` 전체 JSONL 파일 6,183개. 현재 59ms 소요 (백그라운드 스레드). 하지만 파일이 계속 누적되는 구조이므로 시간이 지날수록 악화.

**개선 제안**:
- 프로젝트 디렉토리의 mtime을 먼저 확인하고, 최근에 변경된 디렉토리만 내부 파일을 스캔 (현재 TaskLoader가 이미 이 패턴 사용, line 34: `Date().timeIntervalSince(mtime) < 3600`)
- 또는 이전 스캔 시점 이후 변경된 파일만 재스캔 (마지막 mtime 기억)

### 이슈 5: [LOW] PathDecoder 대시 디코딩이 하이픈 포함 프로젝트명을 파괴

**사용자 시나리오**: `"-Users-anhyobin-dev-mac-app-for-claude"` → `replacingOccurrences(of: "-", with: "/")` → `/Users/anhyobin/dev/mac/app/for/claude` → lastPathComponent = "claude". 프로젝트명이 "mac-app-for-claude"가 아니라 "claude"로 잘린다.

**위치**: `Sources/ClaudeCodeMonitor/Utilities/PathDecoder.swift:7`

**근거**: 현재 이 코드 경로가 트리거되는 조건이 매우 제한적 (JSONL에 cwd 없을 때만). 20개 샘플 모두 cwd 존재 확인. 따라서 실제 영향 LOW.

**개선 제안**: 이 분기 자체를 제거. PathDecoder는 real cwd 경로만 처리하면 됨. 대시 인코딩된 디렉토리명을 역으로 디코딩하는 것은 원본 구분자가 대시인지 경로 구분자인지 구별 불가능하므로 원리적으로 불가.
